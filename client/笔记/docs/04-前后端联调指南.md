# 前后端联调指南

## 1. 联调前检查

确认以下服务都正常：

```text
MongoDB：http://127.0.0.1:27017（默认连接地址）
后端：http://localhost:5000
前端：http://localhost:3000
```

建议后端提供健康检查：

```http
GET /health
```

响应：

```json
{
  "success": true,
  "data": {
    "status": "ok"
  }
}
```

## 2. 先独立测试后端

不要一开始就在 React 页面里排查所有问题。先用 curl 确认接口本身可用。

健康检查：

```bash
curl http://localhost:5000/health
```

创建文章：

```bash
curl -X POST http://localhost:5000/posts \
  -H 'Content-Type: application/json' \
  -d '{
    "title": "第一篇文章",
    "content": "# Hello",
    "author": "前端开发者",
    "tags": ["React", "Node.js"]
  }'
```

获取列表：

```bash
curl http://localhost:5000/posts
```

## 3. 再测试 Vite 代理

前端启动后执行：

```bash
curl http://localhost:3000/api/posts
```

如果直连 `5000/posts` 成功，但 `3000/api/posts` 失败，问题通常在 Vite 代理配置。

当前代理会删除 `/api`：

```text
/api/posts -> http://localhost:5000/posts
```

## 4. 前端调用方式

当前 Axios 封装位于 `client/src/utils/api.ts`：

```ts
const result = await getPosts();

if (result.success) {
  console.log(result.data);
}
```

页面应处理三种状态：

1. loading：请求进行中。
2. success：渲染数据，包括空数组状态。
3. error：展示可理解的错误信息，并允许重试。

推荐的错误处理方式：

```ts
try {
  const result = await getPosts();

  if (!result.success) {
    throw new Error(result.message || '获取文章失败');
  }

  setPosts(result.data);
} catch (error) {
  const message =
    error instanceof Error ? error.message : '网络异常，请稍后重试';
  setError(message);
}
```

## 5. 浏览器 Network 面板怎么看

重点检查：

- Request URL：请求地址是否正确。
- Request Method：方法是否正确。
- Status Code：HTTP 状态码。
- Request Payload：提交的数据。
- Response：后端返回的数据。
- Timing：请求是否超时。

常见现象：

| 现象 | 常见原因 |
| --- | --- |
| `ERR_CONNECTION_REFUSED` | 后端没启动或端口错误 |
| 404 | 前后端路径不一致 |
| 400 | 请求参数未通过校验 |
| 500 | 后端代码或数据库异常 |
| 请求一直 pending | 后端没有结束响应或数据库阻塞 |
| CORS 报错 | 跨域配置错误；开发代理未生效 |
| Axios timeout | 后端超过当前 5 秒超时设置 |

## 6. 当前项目的已知问题

### 根目录启动命令不一致

实际 `client/package.json` 使用：

```json
"dev": "vite"
```

但根目录调用的是 `npm start`，会报缺少脚本。应调用 `npm run dev`。

### 后端启动脚本缺失

根目录会执行 `npm run server`，再进入 server 执行 `npm run dev`，但 server 当前没有 `dev` 脚本。

### 后端源代码缺失

server 当前没有：

- Express 入口
- MongoDB 连接
- Post Model
- 文章路由和 Controller
- 统一异常处理

因此现阶段 API 无法实际响应，需要先完成后端实现。

### 页面尚未调用 API

虽然已有 `utils/api.ts`，但 `home.tsx`、`article.tsx` 和 `publish.tsx` 目前仍使用静态内容，也没有调用对应 API。

## 7. 前后端接口确认清单

联调前双方确认：

- URL 和 HTTP 方法一致。
- 请求字段名称和类型一致。
- 响应结构一致。
- 成功和失败状态码一致。
- 时间、空值和可选字段格式一致。
- 分页参数与分页返回结构一致。
- 登录令牌放在哪里已经约定。
- 错误信息是否可直接展示给用户。

## 8. 安全注意事项

- Markdown 在浏览器渲染时要防止 XSS。
- 服务端必须限制请求体大小。
- 数据库错误详情不要直接返回前端。
- 密码必须哈希，不能明文保存。
- 登录令牌和数据库地址不要写入前端代码。
- 更新和删除文章必须在后端校验权限。
